## Bounded Run

Question: what uncertainty structure is actually earned by the current
finality-trajectory fixtures, and what remains unearned until the repo has
stochastic or protocol-level variation rather than deterministic schedules?

Method:

1. Reuse `TS-PERSONA-SPRINT-001` as the current trajectory family.
2. Extract cross-variant spread for the already-computed finality coordinates.
3. Treat `T62/T64` as the threshold-sensitive comparison lane because they
   already expose verdict flips under declared information thresholds.
4. Separate four uncertainty layers:
   - within-run noise;
   - between-protocol variation;
   - threshold sensitivity;
   - multiscale confidence intervals.

## Work Performed

I audited the existing machine-readable trajectory family in
`results/ts-persona-sprint-001-v0.1.json`.

Across the five existing variants:

- `LOD` is fixed at `0` in all five runs.
- `PG` takes only the scheduled values `5`, `10`, and `15`, with mean `10`.
- micro `DT_max` is fixed at `15` in all five runs.
- holonic `DT_max` is `20`, `25`, or `30`, exactly tracking the hardcoded

- holonic AR(1) accuracy is also always `0.959`.
- micro entropy is always `0.881`.
- holonic entropy is `1.000` for three variants and `0.971` for two variants.
- cointegration is reported `True` in all five variants.

I then audited the detector-side threshold lane in `T64`.

The `results/stern-gerlach-access-window-v0.1.json` threshold sweep shows:

- at threshold `0.75`, observer time `2.5` is finalized;
- at threshold `0.9`, the same time slice is not finalized;
- two observer-time slices flip finalized/not-finalized status across the
  declared threshold family.

This gives the repo one real uncertainty axis that is not just trajectory
schedule variation: threshold-sensitive classification.

## Result

### Main Finding

The current repo does **not** yet earn multiscale confidence intervals for
finality trajectories.

What it currently earns is narrower:

1. **Within-run noise:** not earned. The `TS-PERSONA-SPRINT-001` family is
   deterministic, so repeated runs of the same fixture have zero sampling
   variance by construction.
2. **Between-protocol variation:** partially earned, but only as schedule
   variation. The existing spread in `PG` and holonic dwell is generated by
   changing the hardcoded obstruction window, not by stochastic perturbations,
   observer noise, or inferred latent structure.
3. **Threshold sensitivity:** earned in the narrow detector sense. `T64`
   already demonstrates verdict flips under plausible threshold choices, so
   threshold sensitivity is a live uncertainty source for Q1-facing claims.
4. **Multiscale confidence intervals:** not earned. There is no stochastic
   population, posterior family, or repeated-trial regime from which a
   confidence band over trajectory summaries can be honestly estimated.

### Smallest Honest Uncertainty Table

| uncertainty layer | current status | evidence | interpretation |
| --- | --- | --- | --- |
| within-run noise | absent | deterministic sprint trajectories | no replicate variance exists |
| between-protocol variation | present but weak | `PG in {5,10,15}` | measures schedule family spread, not statistical uncertainty |
| threshold sensitivity | present | `T64` finalized/not-finalized flips across thresholds | real classification fragility |
| multiscale confidence intervals | absent | no stochastic fixture or sampling model | CI language would currently overclaim |

## Interpretation

This run weakens any rhetoric suggesting that the repo already has robust
trajectory statistics.

The present mathematical situation is:

- the trajectory lane provides deterministic derived observables of finite
  schedule families;
- the detector lane provides threshold-fragility witnesses;
- neither lane yet provides a hierarchical uncertainty model connecting
  within-run noise, between-regime variation, and observer-threshold choice.

So the right next step is not to publish confidence intervals on the current
figures. The right next step is to build one stochastic or repeated-trial
fixture where the same declared finality statistic can be estimated across:

- replicate runs within one protocol;
- protocol families;
- threshold choices.

Only then would bootstrap intervals, Bayesian hierarchical summaries, or
multiscale posterior bands become honest rather than decorative.

## Proposed Next Action

Build one bounded uncertainty-ready fixture rather than spreading the request
across the whole repo.

Best next target:

1. start from `T64` because it already has explicit threshold sensitivity;
2. add replicate detector fragments with declared noise draws or perturbation
   trials at fixed observer times; and
3. report three quantities only:
   - within-regime finalized rate;
   - between-threshold flip rate;
   - between-protocol variance in the finalized rate.

That would be the first place where multiscale uncertainty intervals would be
mathematically earned.
