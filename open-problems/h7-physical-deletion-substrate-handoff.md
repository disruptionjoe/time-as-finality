# H7 Physical-Deletion Substrate Handoff

## Status

Issue draft for the thermodynamic-arrow route. This is not H7 evidence, not a
physical-arrow result, and not a thermodynamic derivation.

## Route

Thermodynamic arrow of time.

## Why This Exists

T145, T148, and T152 leave H7 with one narrow live reinstatement gate:

```text
find a physically typed record substrate whose physical-deletion reverse
remains constructor-impossible after full absorber accounting
```

The repo now has good demotion machinery plus a family-level intake screen
([T160](../tests/T160-h7-substrate-family-screen.md)), but it still needs a
concrete handoff for what a serious future substrate proposal must specify.
This file converts that blocker into a reviewable issue draft so H7 is either
made testable or kept dormant as physics.

## Current Strongest Claim

H7 survives only as a reinstatement gate:

```text
No substrate counts as H7 physical-arrow evidence unless it exhibits a
physical_record_deletion reverse that stays constructor-impossible after
free-energy, capacity, sink/export, boundary, provenance, source-copy,
reversible-control, and substrate-specific absorber data are matched.
```

The live question is not whether a record is stable, long-lived, or hard to
erase in practice. The live question is whether deletion is impossible for
substrate-native reasons rather than because of hidden resources, denied
controls, observer boundaries, or idealized limits.

### Minimum Pre-comparison Commitment

Before claiming any H7 relevance, the proposal must freeze:

1. The named physical substrate and encoding of one record token.
2. The write operation, delete operation, and reverse task being compared.
3. The reverse-edge class:
   `physical_record_deletion`, not access loss, support-copy removal, or
   provenance loss.
4. A task-natural future-operation split that deletion topology allegedly
   changes.
5. The fixed absorber vector:
   - erased bits or free-energy floor;
   - blank-capacity change;
   - sink or exported-history state;
   - observer or control boundary;
   - provenance or authority state;
   - source-copy status;
   - reversible-control access.
6. Any substrate-specific absorber data:
   - barrier height and reservoir state for metastability;
   - code distance, syndrome access, and bath/reset channels for protected
     memories;
   - cooling, pumping, or bias-maintenance resources for driven systems;
   - horizon, superselection, or gauge restrictions if claimed.
7. The allowed control class used to judge possibility versus impossibility.
8. A null-control plan covering idealization, hidden sink/export, hidden copy
   handles, denied controls, and merely exponential suppression.

T160 adds a pre-intake rule before detailed review: proposals framed only as
protected memories, driven dissipative states, exact sector restrictions, or
horizon-style inaccessibility should be treated as null by default unless they
already declare why they avoid those family absorbers on a finite operational
reading.

### Required Audit Burden

A serious substrate proposal must later expose enough detail to answer:

1. Is the reverse genuinely `physical_record_deletion`, or only an observer
   boundary or provenance change?
2. Does any apparent impossibility disappear when hidden source-copy
   correlations or reversible controls are restored?
3. Does the proposal rely on changed reservoir, sink, blank-capacity, or
   external work data?
4. Is the reverse impossible, or only kinetically slow, exponentially
   unlikely, or outside one observer's control window?
5. Does the future-operation split persist after all absorber fields are
   matched?
6. Is the impossibility finite and substrate-native, or only an infinite
   barrier, perfect code, exact superselection, or other ideal limit?

If the proposal cannot expose those checks, it does not reopen H7.

## Non-null Outcomes

The only positive near-term outcome is narrow:

```text
A named substrate freezes its deletion task, absorber vector, control class,
and future-operation target, then survives review showing that the reverse is
constructor-impossible after full accounting for substrate-native reasons.
```

That would not prove TaF or derive the full arrow of time. It would only
reopen H7 as a reviewable physical-deletion route.

## Null Outcomes

Treat the route as null for H7 if any of these occur:

- The proposal only shows long lifetime or high but finite barriers.
- Deletion is blocked only because the observer lacks control access.
- Apparent irreversibility disappears when source-copy correlations are
  restored.
- The sink, reservoir, export history, or work budget changes across the pair.
- The claim depends on infinite barriers, exact ideal codes, or other
  constructor idealizations with no finite substrate reading.
- The effect is really access revocation, support-copy removal, or provenance
  loss under another name.
- The future-operation split vanishes once absorber data are matched.
- The proposal never states the allowed control class that defines
  impossibility.

## Decision Rule

If no named substrate can freeze the comparison object above, H7 should remain
paper-facing only as a conditional constructor and resource-accounting audit
discipline.

If a substrate can freeze the object but fails the audit burden, record the
absorber that kills it and keep H7 demoted. Do not promote "hard to erase" or
"protected memory" as arrow evidence by default.

## What This Would Improve

- Converts the remaining H7 blocker into a yes/no substrate ask.
- Prevents practical irreversibility from being confused with
  constructor-impossible deletion.
- Makes future H7 work easier for an external researcher to evaluate.

## What This Weakens

This weakens H7 as an open-ended thermodynamic intuition. The branch now needs
a named substrate with a frozen accounting vector, not another generic story
about stability, barriers, or memory protection.
