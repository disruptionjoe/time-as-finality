# T512: Boundary Adapter Fixed-Source Absorber Gate

## Target Claims

No claim-ledger target. This is a TaF-side finite absorber gate for the T511
boundary-adapter finality packet minimum.

## Setup

T511 requires future boundary-adapter packets to include fixed-richer-source
disclosure as an absorber. T512 makes that requirement explicit.

The packet must predeclare:

```text
Domain
Boundary adapter B
Region / observer boundary
Fixed richer source inventory
Whether the candidate record is contained in that source
Whether predeclared source generators can form the record
Access schedule
Admissible operation/readout class
Candidate record
Undo or readout handle
Ledger or invariant
Positive control
Failure control
Verdict
```

## Success Criteria

- Fixed-source containment plus access-schedule revelation is absorbed as
  disclosure.
- Fixed-source generation is absorbed as a predeclared source-capacity result.
- A nonfixed-source review target is admitted only when the fixed source
  inventory is predeclared and fails to contain or generate the candidate
  record, while the TaF operation/readout handle remains inadmissible and the
  ledger remains conserved.
- Post-hoc source completion, missing source inventory, source-crossing
  language, and cross-repo identity/support shortcuts are rejected or blocked.

## Failure Criteria

- T512 treats a post-hoc source inventory as a valid absorber test.
- T512 treats source-crossing language as TaF finality.
- T512 admits a packet where the undo/readout handle is admissible.
- T512 converts the local absorber gate into GU/TaF/TI identity, support, claim
  movement, public posture, external publication, or cross-repo truth movement.

## Known Physics Constraints

This is finite TaF review machinery. It does not establish a physical boundary
adapter, a source-crossing event, a conserved universal ledger, a hidden central
server, GU support, Temporal Issuance support, or a physics-side irreversibility
claim.

## Contribution Needed

A future packet that wants more than review status must supply a domain-native
adapter theorem, counterexample, or executable fixture whose fixed-source
inventory, operation/readout class, and finality handle are all predeclared
before the witness is chosen.
